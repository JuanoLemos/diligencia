¿Qué es el CBP? el circuito que cuida tu proyecto

El Circuito de Buenas Prácticas (CBP) es el sistema automático que se encarga de cerrar tu sesión de trabajo ordenadamente. Cuando terminás de programar, escribir documentación o arreglar bugs, ejecutás /CBP y el robot analiza qué hiciste, elige el camino correcto, sincroniza los archivos y te prepara el commit. Todo sin que tengas que acordarte de 15 pasos manuales.

Pensalo como "el que cierra la tienda": apaga las luces, barre el piso, deja todo listo para el día siguiente.

¿Cómo funciona paso a paso?

1

Pre-flight — el chequeo rápido 🔍

Antes de hacer cualquier cosa, el sistema revisa que tu proyecto esté al día con la versión de Diligencia. Si hay una versión más nueva, te avisa y te pregunta si querés actualizar. También revisa si hay cambios globales sin documentar. Todo esto sin tocar un archivo.

2

Dispatch — ¿qué camino tomo? 🧭

El sistema mira qué archivos modificaste y elige automáticamente entre tres caminos:

Si solo tocaste...CaminoQué hace
🧑‍💻 Código fuente (src/)commitCommitea rápido, sin tocar documentación. Ideal para bugs o features chicas.
📄 1 a 5 documentosparcialSincroniza los docs, actualiza versiones (patch) y commitea. Liviano.
📚 5+ documentos, guías nuevas, o hitosfullAnálisis completo con agentes, sincroniza todo, puede generar reportes de salud.

También podés forzar un camino específico: /CBP commit, /CBP parcial, /CBP full, /CBP doctor, /CBP version o /CBP updoc.

3

Meta-PLAN — el robot piensa primero 🤖

Antes de modificar cualquier archivo, el sistema lee todo: el estado del proyecto, los archivos que cambiaron, las guías, los ADRs, el CHANGELOG. Arma un resumen completo y te lo muestra. En este punto no se modifica nada — es solo una auditoría.

Después te pregunta: "¿Ejecuto los cambios?" y espera tu confirmación. Esto es clave: nada pasa sin que vos digas que sí.

4

BUILD — la ejecución ⚡

Una vez que aprobaste, el sistema aplica todos los cambios: sincroniza documentación, actualiza índices, genera reportes de salud, versiona, y commitea. Pero ojo: BUILD aplica cambios pero no commitea (salvo que el workflow incluya /version o /commit).

La regla de oro: solo /commit, /CBP y /version pueden ejecutar git commit. Esto evita commits accidentales a mitad de un proceso.

5

Post-workflow — ¿algo más? 🔄

Cuando termina, el sistema puede sugerir pasos adicionales:

  • Si ejecutaste updoc y había correcciones → sugiere /CBP doctor
  • Si ejecutaste version → sugiere /CBP doctor (diagnóstico post-versionado)
  • Si ejecutaste full y quedaron correcciones → sugiere /CBP doctor
  • Si ejecutaste commit sin $REPO definido → te dice cómo pushear manualmente
  • Si ejecutaste parcial → no sugiere nada, ya está todo en orden

Los 6 caminos del CBP

⚡ commit

Cuándo usarlo: Solo tocaste código. Sin docs, sin guías, sin mecánicas.

Qué hace: Pide el mensaje de commit, valida el formato, muestra el diff, commitea y pushea. No toca nada de documentación.

Comando: /CBP commit o simplemente /commit <mensaje>

🔷 parcial

Cuándo usarlo: Tocaste entre 1 y 5 documentos. Cambios livianos.

Qué hace: Auditoría rápida de docs, sincroniza índices, versiona (patch), commitea y pushea. Sin agentes, sin reportes de salud.

Comando: /CBP parcial

🌟 full / completo

Cuándo usarlo: Tocaste 5+ docs, creaste guías nuevas, o estás cerrando un hito del proyecto.

Qué hace: Análisis profundo con 4 workers en paralelo, ejecuta agentes si hace falta, sincroniza todo, genera reporte de salud, versiona y pushea.

Comando: /CBP full o /CBP completo

🔄 updoc

Cuándo usarlo: Querés sincronizar toda la documentación del proyecto al final de una sesión larga.

Qué hace: Escanea todos los docs, detecta stale, actualiza índices, sincroniza guías y mecánicas, versiona, y sugiere doctor si quedaron correcciones.

Comando: /CBP updoc

🩺 doctor

Cuándo usarlo: Querés una revisión completa de salud del proyecto.

Qué hace: Diagnóstico de estructura, código, gaps documentales, temporales, obsoletos, backups. Si encuentra problemas, los corrige y versiona.

Comando: /CBP doctor

📦 version

Cuándo usarlo: Querés cerrar una versión sin sincronizar documentación (ya lo hiciste antes).

Qué hace: Detecta la versión actual, calcula el bump, autogenera el CHANGELOG, hace pre-flight, y si está todo bien, commitea + taguea + pushea.

Comando: /CBP version

Las reglas que siempre aplican

🛑 Regla #17 — Solo 3 comandos commitean

/commit, /CBP y /version son los únicos que pueden ejecutar git commit. Ningún otro comando (ni siquiera BUILD) puede commitear. Los cambios se acumulan en el working tree y vos decidís cuándo sellarlos.

⚠️ Regla #18 — Si hay duda, pausa

Si en cualquier momento el sistema encuentra un estado ambiguo (dos caminos posibles, algo inesperado), se detiene y te pregunta. Nunca asume. Vos tenés el control.

🌍 Regla #19 — Proyectos ajenos no se tocan sin permiso

Si el proyecto NO es Diligencia, el sistema te pide confirmación antes de cualquier acción de git (add, commit, push, tag). Diligencia es la única excepción porque es auto-referencial.

📋 Regla #15 — Pre-flight siempre

Antes de cualquier workflow, el sistema verifica que tu proyecto esté al día con Diligencia. Si no, te avisa y te da la opción de actualizar. Nunca vas a trabajar con una metodología atrasada sin saberlo.

En resumen

CBP es el robot que cierra tu sesión de trabajo. Analiza lo que hiciste, elige el camino correcto, aplica los cambios necesarios, y te prepara todo para que solo tengas que decir "sí, mandale". No importa si tocaste una línea de código o reescribiste toda la documentación — CBP sabe qué hacer.

Flujo típico: trabajás → /CBP → el robot analiza → muestra resumen → vos decís que sí → todo queda listo → /CBP commit → a dormir.
