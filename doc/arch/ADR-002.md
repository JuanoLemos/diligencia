# ADR-002: Sistema de variables para rutas

- **Fecha:** 2026-05-08
- **Estado:** Aceptado
- **Supersedes:** N/A
- **Superseded by:** N/A

## Contexto

Los comandos `.md` contenían rutas hardcodeadas como:
```
/docs/viejo/roadmap.md
```
Al cambiar la estructura del proyecto, todos los comandos se rompían. Cada migración requería editar N comandos.

## Decisión

Introducir un sistema de `$variables` en los comandos. Las variables se definen en `AGENTS.md` bajo la sección `Mapeo de rutas`. Cada variable asigna un nombre a una ruta:

```markdown
## Mapeo de rutas
| Variable | Ruta | Descripción |
|---|---|---|
| $ROADMAP | ROADMAP.md | Roadmap del proyecto |
```

Los comandos usan `$ROADMAP` en lugar de `ROADMAP.md`. Para cambiar la ruta, se edita `AGENTS.md` una sola vez.

## Convención de naming

- Prefijo `$`
- Nombres en MAYÚSCULAS
- Guión bajo como separador: `$RM_TX`, `$BACKEND_DIR`

## Resolución

Cuando el orquestador lee un comando con `$VARIABLE`, busca en `AGENTS.md` la entrada correspondiente y la reemplaza por la ruta real.

Si la variable incluye un `#ancla` (ej. `$RM_TX → ROADMAP.md#tecnico`), el orquestador resuelve el archivo y se posiciona en el ancla.

## Consecuencias

- Positivas:
  - Cambiar una ruta = editar 1 línea en 1 archivo
  - Comandos portables entre proyectos (solo cambia AGENTS.md)
  - Documentación auto-documentada (las variables se explican en AGENTS.md)

- Negativas:
  - Overhead inicial de definir variables
  - El orquestador debe resolver variables correctamente
  - Variables indefinidas producen errores silenciosos

## Ejemplo de migración

**Antes:**
```
1. Leer C:\xampp\htdocs\nemesis\docs\viejos\roadmap.md
```

**Después:**
```
1. Leer $ROADMAP
```

Para cambiar la ruta, editar `AGENTS.md`:
```
| $ROADMAP | ROADMAP.md |
```
